PHP Default Parameters Falsy Values

The Setup

Last night, I lost 2 hours debugging a PHP issue for a project at work. After figuring out what the issue was, I was immediately filled with a rage that burns with the fierce passion of a million suns. At GroupCard, we use our own custom object relational mapping (ORM) framework. In this framework, we have various models that map 1-1 to their respective database tables. All the models extend from a baseModel class, to facilitate the typical CRUD operations. The offending issue was surrounding a boolean default parameter we had in our base model constuctor. Basically, when we create a new model, we pass in parameters that get set on said model’s object instance. Once these parameters are set, we use them to load the rest of the model from our mysql database. Pretty standard ORM. However, we also have a static method available for loading multiple instances of the same model class, called loadAll. This method is used to load an array of model objects containing a common property value. For example:

So in this specific loadAll use case, what we do is instantiate a new object, but don’t immediately load it. Instead, we use this object to create a generic sql statement that will return 1 or more rows from our a database, which we can then use to instantiate an array of objects, as in the getSmiths example.

Where PHP Gets It Wrong

So, in our constructor, I created a $load parameter that defaults to true via PHP’s default parameters syntax. As I stated before, when we’re using loadAll, we don’t want it to attempt to load the object from the database, rather use it to generate sql. So the loadAll method looks something like this:

What we do in this case is pass false as the second argument to the constructor, signaling to not perform the loading of the object. By passing false as the second parameter, the $load variable in the constructor then becomes false and this the loading of the object never happens. This should work, right? Wrong. It turns out PHP interprets default values as “if the passed in value is falsy, then use the default value”. This makes absolutely no sense to me, and it was also causing our query system to break. So when I passed false in to the constructor to disable the automatic loading, PHP evaluated the argument as false, thus using the default value of true.

In fact, you can’t pass any falsy value to a function with a default parameter or you’ll always get the default. In my opinion, the default parameter should only be used if

  • No argument was passed for the paramter, or
  • The argument that was passed for the parameter is NULL

This not only caused me much frustration (and furthered my appreciation for my front end position), but also took me far too long to figure out because I made the unwise assumption that PHP would do it right. Lesson learned.

This (ridiculous) Solution.

The solution to my problem was to simply change the $load parameter to $dontLoad, and have it default to false, as well as change the load block if statement to if ( ! $dontLoad ). I could then pass in true as the second argument to short circuit the object loading.

Ridiculous.

TL;DR

Passing a falsy value to a PHP function with a default value results in always getting the default value.
